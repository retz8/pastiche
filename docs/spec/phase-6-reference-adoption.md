# Phase 6 — Reference adoption `examples/github-primer-react/`

Phase-level spec for the pastiche reference adoption against GitHub Primer. Replaces and restructures the original Phase 6 tasks in `_dev/docs/TODO.md` (6.1–6.7). Locked in grill-me session 2026-05-23.

## Scope

Build a runnable Next.js + `@primer/react` example that serves two purposes simultaneously:

1. **Reference adoption** — demonstrates pastiche from the adopter's perspective; produces the failure-mode artifacts that anchor the project's central claim.
2. **Iteration target** — exercises the plugin in real use; bugs in skills, agents, templates, or scripts discovered here flow back as fixes in this repo.

Four evaluation goals for the phase:

1. Skills/agents work correctly from the adopter's perspective.
2. Pastiche produces UI that faithfully executes the Primer design system.
3. Plugin error handling holds up under edge cases.
4. The FACT extractor covers all scenarios identified during this phase.

Out of scope: extractor field-testing against multiple varied codebases — deferred to a new phase between 6 and 7 (see Follow-ups).

## Locked decisions

### 1. Example name and location

The reference adoption lives at `examples/github-primer-react/` (not `primer-react/`). The `examples/` parent is intentional — future examples follow the same layout.

### 2. Plugin install mechanism: symlink

The example mimics a real pastiche install via a symlink from `examples/github-primer-react/.claude/plugins/pastiche` to the repo-root `dist-plugin/` build output. We cannot publish to a Claude Code plugin marketplace yet, and a symlink keeps the iteration loop tight: a fix in canonical sources → rebuild → the symlink picks it up with no manual re-copy.

### 3. Plugin packaging: `dist-plugin/` build output (new task 6.0)

A thin slice of Phase 7.1 (originally "packaging & docs") is pulled forward into Phase 6 as a new task **6.0**, because the example cannot symlink-install a plugin that doesn't yet exist in loadable shape.

The build produces a `dist-plugin/` directory (gitignored, regenerated by the build script) containing:

- `.claude-plugin/plugin.json` — minimal manifest, generated
- `skills/<name>/SKILL.md` — canonical bodies wrapped with frontmatter (no per-platform adapter; skills ship as a single shape)
- `agents/<name>.md` — canonical bodies wrapped with frontmatter, generated via `adapters/claude-code/agents.template`
- `templates/*` — copied as-is so `$CLAUDE_PLUGIN_ROOT/templates/*` works
- `dist/*.js` — bundled extractor and lint scripts

Phase 8 (formerly 7) later hardens this with marketplace.json, version metadata, README polish, etc.

**Note:** the exact local-install path under `.claude/` on the consumer side (whether Claude Code loads from `.claude/plugins/<name>/` directly or requires a `.claude/settings.json` marketplace pointer) is unverified. Confirm empirically during 6.0/6.1; adjust as needed.

### 4. Build tool: `tsup` for bundling, `tsx` for orchestrator

The build orchestrator is a TypeScript file (`scripts/build.ts`) run via `tsx` (already a dev dependency). It renders adapter templates over canonical bodies, copies `templates/`, generates `plugin.json`, and invokes the bundler.

The bundler is **`tsup`** (esbuild engine, zero config). This revises the bundler choice locked in `docs/spec/phase-4-scripts.md` (which named `bun build`); see Follow-ups for the amendment.

Rationale: bun's only role in v1 was bundling; `tsx` already covers dev-time TS execution; adding a separate runtime (~100 MB) for a single bundling job is not justified at our scale. `tsup` gives esbuild's industry-standard output with zero configuration.

### 5. App shape: narrative GitHub-ish

The example is a small narrative app — three pages, coherent enough that a reader recognizes it as a product surface, not a component swatch sheet:

- Issues list
- Issue detail
- New-issue composer

A real-world-shaped app is required to give pastiche tasks realistic context. The TODO's original "5–8 components" cap is **dropped**: the scaffold and pastiche tasks use whatever Primer components a realistic version of this app actually requires.

### 6. Scaffold-vs-pastiche split: minimum scaffold + many tasks → cherry-pick 3

Task 6.1 hand-scaffolds only the minimum needed to make the app runnable (routing, layout shell, Primer wiring, empty page stubs). Every page body and feature is built via pastiche tasks during 6.5.

The three failure-mode artifacts required by 6.5 (component omission / token omission / wrong choice) are **cherry-picked from many real pastiche invocations**, not designed upfront. If after running a healthy number of tasks (~10) no task naturally surfaces a given failure mode, then a task is designed specifically to fill that gap.

### 7. KNOWLEDGE and WISDOM sourced from primer.style, not from the example app

KNOWLEDGE and WISDOM are curated by walking the GitHub Primer design system website (primer.style), not by reflecting on the example app's codebase. The example app is purely a venue for invoking pastiche.

### 8. KNOWLEDGE/WISDOM pacing: skeleton pass first, deepen just-in-time

A dedicated skeleton pass populates one or two entries in each of the 12 canonical KNOWLEDGE sections and 3–5 atom-tagged WISDOM rules, distilled from the research doc (see §10). `pastiche-lint` must pass after the skeleton pass.

Subsequently, when a pastiche task during 6.5 needs a scenario or atom not yet covered, curation pauses, the relevant primer.style content is reviewed, the missing entry is added, and the task resumes. Pure front-loading is unrealistic; pure JIT keeps the docs incoherent — the hybrid keeps a stable baseline while deepening where real tasks earn it.

### 9. UI quality evaluation: dual bar

Each pastiche-built page is judged on two axes:

1. **Doc compliance** (primary, objective): does the output respect KNOWLEDGE (right atom for the scenario) and WISDOM (no banned combinations)?
2. **Eyeball vs. primer.style** (secondary, subjective): side-by-side against the corresponding primer.style demo for spacing rhythm, hierarchy, layout balance.

When (1) passes but (2) fails, the gap is in KNOWLEDGE/WISDOM itself — that gap is captured as a finding and used to improve curation.

No automated visual diffing, no third-party reviewer.

### 10. Deep research session on primer.style (new task 6.2.5)

A dedicated session walks primer.style end-to-end and produces a human-readable research notebook (plain prose + bullet lists, not pastiche-shaped) at:

`examples/github-primer-react/docs/primer-research.md`

This research doc is **upstream of KNOWLEDGE/WISDOM curation** — it becomes the input that 6.3 and 6.4 distill from. Beyond serving this example, it also serves as prototype evidence for a future "research-from-docs" mode of `pastiche-setup` (see Open items).

### 11. Findings flow

Phase 6 captures bugs, edge cases, and gaps in a single log at:

`docs/phase-6-findings.md`

Each finding gets a one-line entry with description, terminal context (example vs. extractor vs. lint), severity (blocker / nuisance / nice-to-have), and status (open / fixed / promoted-to-TODO). Inline-fixable findings get fixed in the main terminal (edit canonical → rebuild → symlink picks up → re-run failed task in example terminal → mark fixed). Findings too large for an inline fix get promoted to new TODO entries.

Phase 6 closes only when the findings log shows zero open blockers.

### 12. Failure-mode artifact format (6.5)

Artifacts live at `examples/github-primer-react/artifacts/<NN>-<failure-mode>/` (sequenced for pedagogical order, not chronological). Each artifact directory contains:

- `README.md` — what failure mode, why this task, what to look for
- `input.md` — the exact task prompt given to pastiche
- `round1.diff` — implementer round 1 output
- `reviewer.yaml` — reviewer's strict-YAML doubts (verbatim)
- `round2.diff` — round 2 correction or defense
- `final.diff` — what got committed

### 13. Non-affiliation banners (6.6)

A non-affiliation disclaimer appears in:

1. `examples/github-primer-react/README.md` (top of file)
2. `examples/github-primer-react/pastiche/KNOWLEDGE.md` (header)
3. `examples/github-primer-react/pastiche/WISDOM.md` (header)
4. `examples/github-primer-react/docs/primer-research.md` (header)

### 14. Stack pins

- **Primer:** latest stable `@primer/react` at scaffold time, exact-pinned in `package.json` (no `^`) — pastiche correctness depends on FACT matching the installed catalog
- **Next.js:** latest stable, **app router**
- **TypeScript:** yes
- **Styling:** Primer's built-in styling only — no Tailwind, no styled-components, no extra layers
- **Package manager:** npm (matches existing repo tooling)

### 15. Committed artifacts in the example

The example commits `pastiche/config.yaml`, `pastiche/FACT.md`, `pastiche/KNOWLEDGE.md`, and `pastiche/WISDOM.md`. Reproducibility, browsability on GitHub, and reviewability of curation diffs all require this. `dist-plugin/`, `node_modules`, and Next.js build outputs are gitignored.

### 16. Example README contents

In order:

1. Non-affiliation banner (§13)
2. One-paragraph "what this is"
3. Quickstart (install, dev server, note that the plugin is symlinked from the parent repo)
4. Pointers to `docs/primer-research.md`, `pastiche/KNOWLEDGE.md`, `pastiche/WISDOM.md`, `artifacts/`
5. Link back to root `README.md`

### 18. Acceptance criteria for closing Phase 6

## Invariants

- **Symlink, not copy.** All iteration on plugin sources happens in the main repo's canonical files; the example never holds its own edited copy.
- **Built outputs are not committed.** `dist-plugin/` is regenerated on demand and stays gitignored, per the canonical-vs-built discipline.
- **The example is a venue, not a source.** KNOWLEDGE and WISDOM are curated from primer.style, not from the example's codebase.

## Open items

- **Plugin local-install path on the consumer side.** Whether Claude Code loads plugins from `.claude/plugins/<name>/` directly, or requires a local-marketplace pointer in `.claude/settings.json`, is unverified. Confirm during 6.0/6.1.

- **Plugin.json schema details.** The minimum viable `plugin.json` shape (required fields, declaration of skills/agents) for v1 — verify against Claude Code plugin format when implementing 6.0.

- **Expected emergent finding: research-mode `pastiche-setup`.** The existing setup skill assumes adopters can be interviewed about their design system. The 6.2.5 research doc is evidence that some adopters will instead need help *generating* KNOWLEDGE/WISDOM seeds from their own DS documentation. Track during Phase 6 whether this gap is real and severe enough to warrant a setup-skill enhancement; if so, it becomes a new TODO entry in a later phase.
